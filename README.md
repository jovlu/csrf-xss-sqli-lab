Упутство за лабораторијске вежбе из предмета Мрежна безбедност
1. CSRF (Cross-Site Request Forgery)
CSRF (Cross-Site Request Forgery) представља врсту напада на веб апликације у којој нападач наводи аутентификованог корисника да, без свог знања или намере, пошаље захтев ка легитимној апликацији.
Пошто веб прегледач аутоматски шаље сесијске колачиће (cookies) уз сваки захтев ка одговарајућем домену, сервер често није у могућности да разликује легитиман захтев од злонамерног. Као последица, апликација може извршити одређену радњу у име корисника.
Последице CSRF напада могу укључивати:
•	промену лозинке,
•	измену корисничких података,
•	извршавање других осетљивих операција у име жртве.
У овој лаб вежби ћемо демонстрирати напад којим се мења лозинка жртве, тако да жртва не може да се поново логује.
1.1 Припрема окружења
1.	У www folder (C:\wamp64\www) прекопирати све потребне фајлове
2.	Отворити WAMPServer апликацију
3.	Покренути веб прегледач Mozilla Firefox.
4.	Отворити страницу:
about:config
5.	Подесити следећи параметар на вредност false, притиском на стрелице са десне стране:
network.cookie.sameSite.noneRequiresSecure
Ово подешавање омогућава употребу колачића са атрибутом SameSite=None преко HTTP протокола, што је неопходно за демонстрацију CSRF напада у лабораторијском окружењу.
Колачићи могу имати различите SameSite политике:
•	SameSite=None – колачић се шаље уз све захтеве, укључујући cross-site захтеве.
•	SameSite=Lax – колачић се шаље само у ограниченим cross-site сценаријима.
•	SameSite=Strict – колачић се шаље искључиво када захтев потиче са истог сајта.
1.2 Припрема базе података
Пре покретања вежбе потребно је да се креира база података и одговарајућа табела за кориснике, као и да се убаци тест налог. Ово омогућава да апликација има почетне податке над којима ће се демонстрирати CSRF напади током вежбе.
У phpMyAdmin алату, у одељку SQL, покренути следећи SQL код за формирање базе података и табеле са корисником који има корисничко име test и шифру 123 (кликнути на w иконицу доле десно и на Database management па phpMyAdmin):
Користити username: root, и празан password за phpMyAdmin.
CREATE DATABASE IF NOT EXISTS csrf_demo;
USE csrf_demo;

CREATE TABLE IF NOT EXISTS users (
  id INT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
  username VARCHAR(50) NOT NULL UNIQUE,
  password VARCHAR(255) NOT NULL
);

INSERT INTO users (username, password) VALUES ('test','123');

1.5 GET CSRF сценарио
Код GET захтева параметри се налазе у URL-у. Ако апликација омогућава промену стања преко GET метода, нападач може једноставно навести корисника да отвори посебно конструисан URL.
Пример:
http://victim.localhost/CSRF/get/zrtva/index.php?action=change_password&new_password=hacked123
Легитимна страница ће аутоматски послати session cookie, чак и ако је постављен на SameSite=Lax, јер се заправо страница отвара у прегледачу
Покушај напада:
1.	Отворити:
attacker.localhost/CSRF/get/napadac
victim.localhost/CSRF/get/zrtva
2.	Пријавити се на victim страницу.
3.	У Developer Tools проверити session cookie.
4.	Покушати напад кликом на слику на страници нападача.
5.	Излоговати се и покушати поновни логин.
Да ли је лозинка промењена?
Из овог разлога GET захтеви не би требало да мењају стање система.
1.6 Одбрана CSRF токенима
Најефикаснија заштита од CSRF напада је употреба CSRF токена.
CSRF токен је јединствена вредност коју сервер генерише приликом логовања корисника. Токен се чува у форми и шаље уз сваки захтев који мења стање. Сервер проверава вредност токена и одбија захтев ако токен није исправан.
Ток вежбе:

2.	Поново се улоговати (лозинка је hacked123).
3.	Променити лозинку легитимним путем и посматрати URL.
4.	Поново покушати CSRF напад.
5.	Излоговати се и покушати поновни логин.

2. XSS (Cross-Site Scripting)
XSS (Cross-Site Scripting) представља врсту напада на веб апликације у којој нападач убацује злонамерни JavaScript код у садржај странице, који се затим извршава у прегледачу других корисника. До напада долази када апликација не обрађује на адекватан начин кориснички унос, па се злонамерни садржај приказује као део легитимне странице.
У оквиру ове лабораторијске вежбе обрађују се три основне врсте XSS напада:
•	Reflected XSS
•	Stored (Persistent) XSS
•	DOM-based XSS
Reflected XSS је тип XSS напада код којег се злонамерна скрипта не чува на серверу, већ се одмах „рефлектује“ у HTTP одговору апликације, најчешће кроз URL параметре или вредности послате путем форме.
Stored (Persistent) XSS представља тип XSS напада у којем се злонамерна скрипта трајно чува на серверској страни апликације, најчешће у бази података, на пример у коментарима или порукама. Такав садржај се аутоматски извршава у прегледачу сваког корисника који приступи компромитованој страници, због чега је овај тип напада посебно опасан.
DOM-based XSS је тип XSS напада који се извршава искључиво на клијентској страни, без директног учешћа сервера. До напада долази када JavaScript код у прегледачу користи непроверен или несаниран кориснички унос, на пример из URL-а или DOM-а, и динамички га убацује у страницу, чиме се омогућава извршавање злонамерног кода.
2.1 Припрема базе података
Пре почетка XSS вежби потребно је да се креира база података и одговарајуће табеле које ће се користити у различитим XSS сценаријима. Једна табела служи за бележење украдених података у reflected XSS примеру, друга за чување коментара у stored XSS примеру, а трећа за бележење података у DOM-based XSS сценарију.
У phpMyAdmin алату покренути следећи SQL код:
CREATE DATABASE IF NOT EXISTS xss_demo;
USE xss_demo;
CREATE TABLE IF NOT EXISTS stolen_reflected (
id INT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
stolen TEXT NOT NULL,
created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
CREATE TABLE IF NOT EXISTS comments (
id INT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
author VARCHAR(100),
body TEXT,
created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
CREATE TABLE IF NOT EXISTS stolen_dom (
id INT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
stolen TEXT NOT NULL,
created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

1.	Отворити страницу localhost/XSS/reflected.php.

Овим сценаријом се показује да reflected XSS може омогућити извршавање скрипте унутар легитимне странице, као и слање осетљивих података ка нападачу.

5.	Освежити страницу и уочити ефекат унетог садржаја.
6.	Унети још неколико злонамерних коментара, пример: <style>body{background:red}</style>
7.	Применити заштиту у коду (XSS/stored.php) и поновити напад.
Објасните напад и како заштита у коду функционише.
Овај део вежбе показује да је stored XSS посебно опасан зато што се злонамерни садржај трајно чува у бази и извршава за све кориснике који приступе страници.
2.4 Ток вежбе — DOM-based XSS
У овој вежби, користићемо три странице: dom.html, dom-attacker.html и dom-stolen.php. Први је легитимна страница, други служи да прикаже малициозан линк ка легитимној страници, као нпр. малициозан e-mail, трећи служи да прикаже украдене податке са легитимне странице.
У прва два корака приказујемо да се напомене на страници додају преко URL-a. Ову могућност нападач користи да украде од корисника тајне податке и смести их у базу. Потом се ови украдени подаци  из базе приказују на страници  dom-stolen.php  да би се демонстрирала крађа.
1.	Отворити страницу http://localhost/XSS/dom.html.
2.	Отворити следећи URL:
http://localhost/XSS/dom.html#Primer: Ovako mozete da unosite podatke%20<strong>Kroz URL</strong> bez komunikacije sa serverom.
3.	Освежити страницу и посматрати промену приказаног садржаја.
4.	Уочити да овај део URL-а не стиже до сервера, већ се обрађује искључиво на клијентској страни.
5.	Затворити таб и поново отворити страницу http://localhost/XSS/dom.html.
6.	У другом табу отворити страницу http://localhost/XSS/dom-stolen.php.
7.	Отворити страницу http://localhost/XSS/dom-attacker.html и кликнути на линк.
8.	Проверити да ли је дошло до слања података ка страници dom-stolen.php.
9.	Применити заштиту у коду (XSS/dom.html) и поновити напад.
Објасните напад и како заштита у коду функционише.
Овим сценаријом се показује да злонамерна JavaScript логика може бити садржана искључиво у URL-у и да се напад у потпуности може реализовати на клијентској страни.
3. SQL Injection
SQL Injection представља напад у којем нападач убацује злонамерне SQL команде у кориснички унос који се директно користи у SQL упиту.
Последице могу бити:
•	заобилажење аутентификације
•	приступ осетљивим подацима
•	измена или брисање података
•	компромитација базе података
3.1 Припрема базе података
У phpMyAdmin унети:
CREATE DATABASE IF NOT EXISTS sqli_demo;
USE sqli_demo;

CREATE TABLE IF NOT EXISTS users (
id INT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
username VARCHAR(255) NOT NULL,
password VARCHAR(255) NOT NULL
);

INSERT INTO users (username, password)
VALUES ('ana', 'lozinka');
